# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

**Loops** ‚ÄĒ a multi-loop autonomous-agent platform. Each "loop" is a recurring, headless Claude Code agent for one domain (SEO, dead-code cleanup, webview refactor, ‚Ä¶). A loop's **orchestrator** discovers work and fans it out; **workers** each implement one item and open a PR. **Humans merge** ‚ÄĒ the engine never merges, deploys, or force-pushes.

The engine is shared; a loop differs only in its `mission.md` (what to find, how) and `config.json` (repo / Linear / schedule / concurrency).

There is **no build, test, or lint step**. The engine is **zero dependencies** ‚ÄĒ `zsh` scripts + Node built-ins (`http`, `child_process`, `fs`); don't go looking for `package.json`, a test runner, or CI for this repo; there is none. **One exception, dashboard UI only:** the dashboard vendors [Oat UI](https://oat.ink) (MIT, ~10KB) as pinned static assets in `vendor/oat.min.{css,js}` ‚ÄĒ served by `dashboard-server.mjs` at `/vendor/*` and loaded by `dashboard.html`. **This is still no-build** (edit `dashboard.html` ‚Üí refresh; no bundler, no `npm install`, `dashboard-server.mjs` stays Node-builtin-only). To bump Oat, re-`curl` a pinned version from `unpkg.com/@knadh/oat@<ver>/` into `vendor/` (Oat is sub-v1 ‚ÄĒ pin, don't float).

## Hard prerequisite: cmux

The platform is built on **cmux** (a terminal with socket control). The dispatcher and dashboard run *inside cmux panels*, and workers spawn as *live cmux tabs*. Non-cmux environments are unsupported. `claude`, `gh`, `node`, `git` must also be present.

## Commands

```sh
./install.sh                 # detect tools ‚Üí write loops.env, ~/.loops-home, symlink create-loop skill
./loopctl dashboard          # dashboard at http://localhost:8422 (opens a cmux panel)
./loopctl start              # start the global dispatcher (cmux panel)
./loopctl stop|pause|resume|status
./loopctl run-now <loop-id>  # fire one orchestrator cycle for a loop immediately
```

There are no unit tests. To exercise a change, run a single orchestrator cycle directly:

```sh
LOOP_MODE=full ./bin/run-once.sh <loop-id>      # full cycle (headless claude -p), logs ‚Üí loops/<id>/state/run.log
LOOP_MODE=reconcile ./bin/run-once.sh <loop-id> # only merge-cleanup + snapshot (fast)
node ./bin/render-prompt.mjs <loop-id> orchestrator   # inspect the rendered prompt without running it
```

`LOOP_MODE` is `full` (default) | `audit_only` (skip fan-out) | `reconcile` (only STEP 1 merge-cleanup + STEP 4 snapshot) | `retro` (performance analysis ‚Üí update `state/learnings.md`; uses `retro-base.md`, no discovery/fan-out).

## Architecture

The runtime is a chain of processes, each launched in a fresh/empty context. **All cross-run continuity lives in the Linear ledger, not in memory.**

```
dispatch.sh (cmux panel, loops while-true)
  ‚ĒĒ‚ĒÄ per enabled loop, on schedule ‚Üí spawn-orchestrator.sh <id>   (FOREGROUND ‚ÄĒ must keep cmux socket access)
       ‚ĒĒ‚ĒÄ run-once.sh <id>   (claude -p <rendered orchestrator prompt>, headless, in the loop's orchestrator worktree)
            ‚ĒĒ‚ĒÄ orchestrator fans out ‚Üí spawn-worker.sh <id> <issue>   (one per Backlog item, up to cap)
                 ‚ĒĒ‚ĒÄ new cmux tab runs worker-run.sh ‚Üí claude <rendered worker prompt>   (live TUI, dedicated worktree+branch)
                      ‚ĒĒ‚ĒÄ worker implements 1 issue ‚Üí /gbase:go (polish+PR) ‚Üí preview check ‚Üí STOP (no merge)
```

- **`bin/_common.sh`** ‚ÄĒ sourced by every script. Derives `LOOPS_HOME` from its own location (`bin/`'s grandparent) and sources `loops.env`. This is why the repo works wherever it's cloned: paths are dynamic, never hardcoded.
- **`bin/render-prompt.mjs`** ‚ÄĒ the templating core. Combines `bin/{orchestrator,worker,verifier,retro}-base.md` (the shared engine logic) + the loop's `mission.md` + `state/learnings.md` (`{{LEARNINGS}}`, retro-accumulated lessons) + `config.json` values, substituting `{{VAR}}` tokens. Editing engine behavior = editing the `-base.md` files, not the per-loop files.
- **Feedback layer (closes the loop; all opt-in via config, all deterministic-guarded):**
  - **`bin/rework-worker.sh`** ‚ÄĒ when a human reviewer leaves `CHANGES_REQUESTED` on an open PR **or** the verifier records a ‚ĚĆ fail verdict, respawns the worker in its **preserved worktree** (`LOOP_REWORK=1` ‚Üí worker-run injects a rework block: address explicit requests only, non-force push, reply to comments, no merge-wait). Orchestrator STEP 1 calls it for every OPEN PR unconditionally ‚ÄĒ 3 built-in guards make that safe: new-feedback gate (won't refire until a review/verdict *newer* than the last rework), per-issue cap `LOOP_REWORK_MAX` (default 2, then `rework-exhausted` surfaces to dashboard/bot), live-tab dedup. State: `state/rework.json`.
  - **`bin/spawn-verifier.sh` + `verifier-run.sh` + `verifier-base.md`** (config `"verify": true`, pr-mode only) ‚ÄĒ maker/checker split: after the worker opens a PR it spawns a **fresh-context verifier** in a detached `-vf` worktree (ūüĒé tab) that grades the PR against the issue's acceptance criteria, actually runs repo checks, and posts an evidence-backed verdict (‚úÖ pass / ‚ö†ÔłŹ concerns / ‚ĚĆ fail) to PR + Linear + `state/verify/<ISSUE>.json`.   - **`vision.md`** (per-loop, optional, human-owned like `mission.md`) ‚ÄĒ product direction (target user / north star / core assets / non-goals) injected as `{{VISION}}` into orchestrator¬∑retro¬∑validator prompts. ## A loop's anatomy

`loops/<id>/` (gitignored ‚ÄĒ user data) contains:
- `config.json` ‚ÄĒ schema: `id, name, emoji, repo, baseRef (origin/develop), prBase (develop), branchPrefix, orchestratorWorktree, worktreePrefix, linearProjectId, linearProjectUrl, maxWorkers, backlogTarget, schedule{startAt,intervalSec}, enabled` + opt-in feedback-layer fields: `verify (bool), budget{dailyUsd}, on{ciFailure,prReview,linearNew}, retro{everyCycles}`. See `examples/*/config.json` and the README field table.
- `mission.md` ‚ÄĒ injected as `{{MISSION}}` into the orchestrator prompt; defines *what work to discover and how* (the only domain-specific logic).
- `state/` ‚ÄĒ runtime: `snapshot.json` (dashboard reads this ‚ÄĒ orchestrator writes it in STEP 4), `runs.jsonl` (append-only event feed), `run.log`, `next_fire`, `decisions/<ISSUE>.md` (human-gate resolutions), `PAUSED`, `costs.jsonl` (headless-run usage), `rework.json` (review-rework counters), `events.json` (event-poll cursors), `verify/<ISSUE>.json` (verifier verdicts), `learnings.md` (retro lessons ‚Üí `{{LEARNINGS}}`), `liveness.json` (watchdog).

The Linear project **is** the state machine (dedup across runs): `Backlog ‚Üí In Progress ‚Üí In Review ‚Üí Done/Canceled`. A "run log" tracking issue per loop gets one comment per cycle.

## Conventions that matter when editing

- **Never merge / deploy / force-push** from engine code or prompts. Workers open PRs only; merging is the human gate. Preserve this in any prompt or script change. **Exception (opt-in):** a loop with `"delivery": "direct"` in its `config.json` makes its workers push straight to `prBase` (non-force, rebase-then-push, stop on conflict) instead of opening a PR ‚ÄĒ used for personal/no-reviewer repos like the `loops-improve` self-improvement loop. Default (no `delivery` field) = `"pr"` = the safe PR flow. `render-prompt.mjs` swaps the worker's step 4‚Äď7 (`{{WORKER_DELIVERY}}`) and an orchestrator note (`{{DELIVERY_NOTE}}`) based on this flag. **force-push stays banned in both modes.**
- **No silent fallback.** Both base prompts forbid `?? default`, swallowed catches, and guessing ‚ÄĒ fix root cause. The target repos enforce this via their own CLAUDE.md/AGENTS.md (which workers/orchestrators must obey, since cwd is the *target* repo's worktree, not this one).
- **`spawn-orchestrator.sh` must stay foreground** (the caller backgrounds it). Detaching loses cmux socket access, so workers can't spawn ‚ÄĒ hence the repeated "FOREGROUND / cmux žēąžóźžĄú žč§ŪĖČ" warnings.
- **cmux lazy-PTY: a tab's command only starts once the tab actually renders on screen.** `new-workspace --command` merely queues "text+Enter"; from a non-cmux caller (launchd, an external shell) ‚ÄĒ or into a window that's minimized / on another Space ‚ÄĒ the workspace object is created but no PTY exists, so the command silently waits and fires **much later** when the tab first renders (the double-dispatcher hazard). Therefore every panel spawn goes through **`bin/spawn-panel.sh`**: create ‚Üí verify materialization (`read-screen`) ‚Üí escalate (select ‚Üí app activate ‚Üí new-window+move+focus) ‚Üí on final failure **close the workspace** (so nothing fires late) and exit non-zero. `loopctl start|dashboard|bot|run-now` all use it. This constraint is also why supervisor `full` revives the dispatcher via the dashboard proxy first, and why the `panels` scope runs from inside dispatch.sh rather than launchd.
- **PR URLs come only from `gh pr view --json url`** ‚ÄĒ never construct `org/repo` URLs by hand (origin may be a GitHub mirror).
- Concurrency: `in-flight = (In Progress) + (In Review)`; cap = `min(maxWorkers, LOOP_MAX_WORKERS)`. Per-loop run lock is a `/tmp/loop-<id>.lockdir` mkdir.
- **human-gate**: an issue whose body says "human-gate" is *not* implemented by a worker ‚ÄĒ it surfaces to the dashboard (ūüĒī) for a human decision, which is written to `state/decisions/<ISSUE>.md` and injected back into the worker as an authoritative override on the next spawn.
- **Terminal-state cleanup.** Once an issue reaches a terminal state ‚ÄĒ `Done` (PR merged) or `Canceled` (PR closed / user discarded) ‚ÄĒ its cmux tab + git worktree + branch are removed automatically. This runs deterministically in `bin/run-once.sh` *after* the headless orchestrator (every cycle, all `LOOP_MODE`s) via `bin/cleanup-issue.sh <loop> <issue>` ‚ÄĒ idempotent, matches the cmux tab by title (`ūüõ†/‚Ü© <loop> <issue>`), gated on the worktree dir actually existing. The same helper backs the dashboard `cancel-issue` ("ūüóĎ Ž≤ĄŽ¶¨Íłį") and a per-issue "ūüßĻ ž†ēŽ¶¨" button; `bin/cleanup-loop.sh` (dashboard `delete-loop` and `loopctl cleanup <loop>`) sweeps a whole loop. **In-progress worktrees (In Progress / In Review) are still preserved** so a user can `claude --resume` them; the LLM prompts never do this cleanup (deterministic shell only); **force-push / merge / deploy stay banned.** The orchestrator always rebuilds its own worktree from `baseRef` (never the user's working tree).
- **Liveness & stall recovery is deterministic, keyed on Linear (not the stale snapshot).** Two shell loops run in `dispatch.sh` every ‚Č§60s (both skip a loop while its `/tmp/loop-<id>.lockdir` is held, so they never race an orchestrator run):
  - **`bin/watchdog.sh`** ‚ÄĒ the in-flight set is **Linear `started`** (always fresh; the old code keyed on existing-worktrees + hourly `snapshot.json`, which missed no-worktree ghosts and lagged a full cycle). ‚ö†ÔłŹ A cmux tab does **not** reliably die with its `--command` ‚ÄĒ the pane can drop to a bare shell, and a cmux **app restart session-restores title-only shell tabs** (dead worker looks alive ‚Üí wedged false-positive + heal blocked). So liveness is process-based: `worker-run.sh` writes `state/live/<ISSUE>.pid` on start, and on exit removes it + retitles its tab **‚ŹĻ** (out of the `ūüõ†|‚Ü©` liveness regexes; cleanup-issue/loop sweep ‚ŹĻ too). Per `started` issue: *live tab* ‚Üí first, if a pidfile exists and its pid is dead, the tab is a corpse ‚Üí watchdog closes it and lets the next pass heal (no pidfile ‚Üí conservative, legacy behavior); else `cmux read-screen` hash; if the screen is frozen ‚Č• `LOOP_WEDGE_SEC` (300s) it's **`wedged`** (surfaced only ‚ÄĒ never auto-killed, to avoid killing a slow-but-fine worker). *No tab* ‚Üí if a branch **open PR** exists (pr-mode) it's In Review (done, awaiting human merge ‚Üí leave it); else it's a dead In Progress ‚ÄĒ if a **worktree remains** (progress) ‚Üí `bin/heal-worker.sh` resumes it in place (grace ‚Üí `LOOP_HEAL_MAX` attempts ‚Üí `escalated`), if **no worktree** it's a ghost the watchdog does **not** re-spawn (would bypass the cap).
  - **`bin/cleanup-terminal.sh` (the reaper)** ‚ÄĒ beyond terminal cleanup, when Linear is fresh it also (4) reverts **no-worktree `started` ghosts ‚Üí Backlog** via `bin/linear-move.mjs` (a ghost pins `in-flight` so `capacity = cap ‚ąí in-flight` goes ‚Č§0 and the orchestrator stops spawning ‚ÄĒ this is the usual "loop silently stops"; reverting frees the slot so the orchestrator re-spawns within cap), and (5) reclaims **Backlog-issue worktree litter** (dead worker reverted to Backlog, no live tab/PR) via `cleanup-issue.sh`. Both gated on live Linear + no open PR; `escalated` issues are vetoed. **Backlog moves are not merge/cancel ‚ÄĒ the no-merge rule is intact.**
  - The dashboard reads `state/liveness.json` and surfaces `stuck` (escalated), `wedged`, and `healing`. The orchestrator prompt now **defers** dead-worker handling to these loops (LLM is a rare backstop), so it no longer races them.
