# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

**Loops** — a multi-loop autonomous-agent platform. Each "loop" is a recurring, headless Claude Code agent for one domain (SEO, dead-code cleanup, webview refactor, …). A loop's **orchestrator** discovers work and fans it out; **workers** each implement one item and open a PR. **Humans merge** — the engine never merges, deploys, or force-pushes.

The engine is shared; a loop differs only in its `mission.md` (what to find, how) and `config.json` (repo / Linear / schedule / concurrency).

There is **no build, test, or lint step**. The engine is **zero dependencies** — `zsh` scripts + Node built-ins (`http`, `child_process`, `fs`); don't go looking for `package.json`, a test runner, or CI for this repo; there is none. **One exception, dashboard UI only:** the dashboard vendors [Oat UI](https://oat.ink) (MIT, ~10KB) as pinned static assets in `vendor/oat.min.{css,js}` — served by `dashboard-server.mjs` at `/vendor/*` and loaded by `dashboard.html`. **This is still no-build** (edit `dashboard.html` → refresh; no bundler, no `npm install`, `dashboard-server.mjs` stays Node-builtin-only). To bump Oat, re-`curl` a pinned version from `unpkg.com/@knadh/oat@<ver>/` into `vendor/` (Oat is sub-v1 — pin, don't float).

## Hard prerequisite: cmux

The platform is built on **cmux** (a terminal with socket control). The dispatcher and dashboard run *inside cmux panels*, and workers spawn as *live cmux tabs*. Non-cmux environments are unsupported. `claude`, `gh`, `node`, `git` must also be present.

## Commands

```sh
./install.sh                 # detect tools → write loops.env, ~/.loops-home, symlink create-loop skill
./loopctl dashboard          # dashboard at http://localhost:8422 (opens a cmux panel)
./loopctl start              # start the global dispatcher (cmux panel)
./loopctl stop|pause|resume|status
./loopctl run-now <loop-id>  # fire one orchestrator cycle for a loop immediately
```

There are no unit tests. To exercise a change, run a single orchestrator cycle directly:

```sh
LOOP_MODE=full ./bin/run-once.sh <loop-id>      # full cycle (headless claude -p), logs → loops/<id>/state/run.log
LOOP_MODE=reconcile ./bin/run-once.sh <loop-id> # only merge-cleanup + snapshot (fast)
node ./bin/render-prompt.mjs <loop-id> orchestrator   # inspect the rendered prompt without running it
```

`LOOP_MODE` is `full` (default) | `audit_only` (skip fan-out) | `reconcile` (only STEP 1 merge-cleanup + STEP 4 snapshot) | `retro` (performance analysis → update `state/learnings.md`; uses `retro-base.md`, no discovery/fan-out).

## Architecture

The runtime is a chain of processes, each launched in a fresh/empty context. **All cross-run continuity lives in the Linear ledger, not in memory.**

```
dispatch.sh (cmux panel, loops while-true)
  └─ per enabled loop, on schedule → spawn-orchestrator.sh <id>   (FOREGROUND — must keep cmux socket access)
       └─ run-once.sh <id>   (claude -p <rendered orchestrator prompt>, headless, in the loop's orchestrator worktree)
            └─ orchestrator fans out → spawn-worker.sh <id> <issue>   (one per Backlog item, up to cap)
                 └─ new cmux tab runs worker-run.sh → claude <rendered worker prompt>   (live TUI, dedicated worktree+branch)
                      └─ worker implements 1 issue → /gbase:go --no-review (polish+PR) → preview check → resident /gbase:monitor watch until merge/close (unattended rules; still never merges)
```

- **`bin/_common.sh`** — sourced by every script. Derives `LOOPS_HOME` from its own location (`bin/`'s grandparent) and sources `loops.env`. This is why the repo works wherever it's cloned: paths are dynamic, never hardcoded.
- **`bin/render-prompt.mjs`** — the templating core. Combines `bin/{orchestrator,worker,verifier,retro}-base.md` (the shared engine logic) + the loop's `mission.md` + `state/learnings.md` (`{{LEARNINGS}}`, retro-accumulated lessons) + `config.json` values, substituting `{{VAR}}` tokens. Editing engine behavior = editing the `-base.md` files, not the per-loop files.
- **Feedback layer (closes the loop; all opt-in via config, all deterministic-guarded):**
  - **Resident monitor (pr-mode default).** The worker does **not** stop after opening its PR: after preview check / Linear In Review / verifier spawn, it stays in `/gbase:monitor` until the PR is MERGED/CLOSED, auto-applying mechanical CI fixes and mechanical/explicit review comments in-session (non-force push only). The worker prompt (`render-prompt.mjs` `WORKER_DELIVERY_PR` step 7) overrides the skill's interactive behavior for unattended tabs: no merge/approve/close, **no `--force-with-lease`** (conflicts resolved only via non-force `git merge origin/<prBase>`, otherwise surfaced as comments), **no AskUserQuestion** (ask-gate situations → PR reply + Linear "🚧 사람 판단 필요" comment, keep watching), `--no-review` (monitor's self-review pass is AskUserQuestion-gated; the verifier is the loops-native checker), and no tab retitle (🛠/↩ is the engine's liveness signal). The gbase skills themselves are never modified. Watchdog's DELIVERED gate means a PR-open issue is never wedge-checked, so the idle watch causes no false positives; live-TUI cost of the resident session is unmeasured (same scope as workers). If the session dies (usage limits, cmux restart), the paths below take over — graceful degradation to the pre-resident behavior.
  - **`bin/rework-worker.sh`** — when a human reviewer leaves `CHANGES_REQUESTED` on an open PR **or** the verifier records a ❌ fail verdict, respawns the worker in its **preserved worktree** (`LOOP_REWORK=1` → worker-run injects a rework block: address explicit requests only, non-force push, reply to comments, then re-enter the resident monitor watch). With the resident monitor alive this never fires (live-tab dedup) — it is the **dead-session fallback** that also re-establishes the watcher. Orchestrator STEP 1 calls it for every OPEN PR unconditionally — 3 built-in guards make that safe: new-feedback gate (won't refire until a review/verdict *newer* than the last rework), per-issue cap `LOOP_REWORK_MAX` (default 2, then `rework-exhausted` surfaces to dashboard/bot), live-tab dedup. Orchestrator STEP 1's direct mechanical-CI-fix is gated on the same signal: it only touches the branch when rework's output shows no live tab (avoids racing the resident session). State: `state/rework.json`.
  - **`bin/spawn-verifier.sh` + `verifier-run.sh` + `verifier-base.md`** (config `"verify": true`, pr-mode only) — maker/checker split: after the worker opens a PR it spawns a **fresh-context verifier** in a detached `-vf` worktree (🔎 tab) that grades the PR against the issue's acceptance criteria, actually runs repo checks, and posts an evidence-backed verdict (✅ pass / ⚠️ concerns / ❌ fail) to PR + Linear + `state/verify/<ISSUE>.json`. The verifier **cannot edit code** — Edit/Write are blocked via `--disallowedTools` (structural, not prompt). fail → rework path above. Never merges/moves Linear states.
  - **`bin/spawn-validator.sh` + `validator-run.sh` + `validator-base.md`** (config `"validate": true`; proposal-type loops — see `docs/factory.md` M2) — discoverer/skeptic split: after each orchestrator run, `run-once.sh` deterministically spawns a **fresh-context validator** (detached `-vd` worktree at `baseRef`, 🧪 tab) for every un-judged human-gate proposal issue in the snapshot. It re-verifies the proposal's evidence for real (WebFetch reproduction, repo grep, competitor URLs), runs a distilled YC-style demand diagnostic (specificity / interest≠demand / status-quo-as-competitor / wedge / distribution), challenges premises against `vision.md`, must propose one cheaper alternative, and records a verdict (🟢 strengthen / 🟡 narrow / 🔴 reject) to Linear + `state/validate/<ISSUE>.json` — surfaced next to `gate.ask` in the dashboard gate modal and as a 🧪 badge. Edit/Write blocked structurally; never approves/rejects (the human gate does), never moves Linear states. Skips if a verdict or a human decision (`decisions/<ISSUE>.md`) already exists.
  - **`vision.md`** (per-loop, optional, human-owned like `mission.md`) — product direction (target user / north star / core assets / non-goals) injected as `{{VISION}}` into orchestrator·retro·validator prompts. Split from mission so retro can never touch direction; editable in the dashboard ⚙️ modal (`save-vision`, `GET /api/mission?vision=1`). Empty/absent → block omitted entirely.
  - **`bin/event-poll.sh`** (config `on:{ciFailure,prReview,linearNew}`) — dispatch.sh polls ≤60s and pulls `next_fire` to now on: new CI failure on `prBase`, new human review on loop PRs, new Backlog issue in the Linear project. No new execution path — firing stays in the dispatcher's existing loop; every trigger is recorded in `runs.jsonl` (`event:"trigger"`). Cursors in `state/events.json` (first poll seeds silently).
  - **`bin/record-cost.mjs`** — every headless run (`run-once.sh` cycles, verifier) executes claude with `--output-format json`; this splits the output into the human-readable `run.log` text + a `state/costs.jsonl` usage record. `dashboard-server.mjs` aggregates it as per-loop `economics` (today/total USD, cost per merged, merge-rate, rework count); config `budget:{dailyUsd}` is a **soft cap** — dispatcher skips fires while today's sum ≥ cap (in-flight workers untouched, resets at midnight). Live-TUI worker cost is *not* measured (headless-only scope, labeled as such in the UI) — this now includes the resident monitor phase, which extends each worker session's unmeasured lifetime until merge.
  - **Self-healing platform layer (process-level — closes the "engine itself died" gap).** All other recovery (watchdog/reaper/self-update/event-poll) lives *inside* `dispatch.sh`'s while-loop, so the dispatcher itself was a single point of failure. Three deterministic-shell pieces close it (no merge/deploy/force-push anywhere):
    - **`bin/supervisor.sh`** — one watchdog pass, two scopes split by the cmux render constraint (below). `full` (launchd agent `com.loops.supervisor`, 60s; `loopctl supervisor install|remove|status|run`) supervises **the dispatcher only**: dead → revive via ⓪ reuse an already-queued **`⏳`-marked** 🔁 tab (only the intentional-queue marker counts — a marker-less un-materialized 🔁 tab may be a session-restored shell whose "reuse" is a no-op; a materialized-but-still-⏳ tab is a fired-and-died shell → closed) ① dashboard `/api/control start` proxy ② direct `loopctl start`, then verify the pid ~3s later. **Death vs spawn-blocked**: a "restart" only counts toward crash-loop when a real death is observed (`dispatcher.lastAlivePid` seen alive then gone, or new `dispatcher start (pid` lines in dispatcher.log beyond `dispatcher.startsSeen`); if neither signal fires the dispatcher never started (backgrounded cmux queue) → **spawn-blocked**: retry without counting/hold + rate-limited "foreground cmux" Telegram hint — this is what previously mis-escalated to a 30-min hold that blocked recovery. `panels` (called from dispatch.sh housekeeping ≤60s) supervises **dashboard + bot** from the dispatcher's PTY context, and runs **`sweep_panels`**: each panel records its own tab at boot via `cmux identify` (`state/panel.{dispatcher,dashboard,bot}.ref`), and the sweep closes every 🔁/📊/🤖-titled tab that isn't the recorded live one (plus all `⏹ loops *` exit-marked tabs) — this is what reaps the title-only shells a cmux app restart session-restores (worker tabs have their own pidfile/watchdog path; `🤖 loop builder` and user tabs are untouchable; missing ref file or empty list-workspaces → that class is skipped, flake principle). Intent markers `state/STOPPED.{dispatcher,bot}` (written by `loopctl stop` / dashboard stop actions, cleared by every start path) distinguish crash from deliberate stop — marked components are never revived. Dashboard "process alive but HTTP dead" → surfaced only, never killed (wedged philosophy). Actions append to `state/supervisor-events.jsonl`; alerts go out via **`bin/tg-notify.mjs`** (direct Telegram send, deliberately *not* through the bot — the bot is itself a supervised component), rate-limited per key. State: `state/supervisor.json`; opt-outs: `LOOPS_SUPERVISE_DASHBOARD=0`.
    - **Dispatcher-only queued-spawn exception** — `spawn-panel.sh` normally discards a workspace that fails to materialize (no-late-fire rule). With `SPAWN_PANEL_QUEUE_OK=1` (used *only* for the dispatcher by `loopctl start` and the dashboard `start` action) it instead leaves ONE tab queued, titled `🔁 loops dispatcher ⏳`, and exits 2: while the cmux app is backgrounded **no** new tab renders (escalation included), so discard-and-retry can never converge — the ⏳ tab auto-fires the moment the user foregrounds cmux, and the dispatch.sh double-start guard makes any duplicate firing harmless (dispatch.sh strips the ⏳ marker and records `state/panel.dispatcher.ref` + runs one immediate panels sweep at startup, and retitles its tab `⏹ loops dispatcher` on clean exit). Never use QUEUE_OK for workers/verifiers — they have no double-start guard.
    - **Crash-loop guard + self-update rollback** — ≥`LOOPS_CRASHLOOP_N`(3) dispatcher deaths within `LOOPS_CRASHLOOP_WINDOW`(600s): if the newest `state/.update_history` line (appended by self-update.sh on every ff) is <`LOOPS_ROLLBACK_WINDOW`(1h) old and HEAD == its result, the update is the suspect → `git reset --keep <prev-sha>` (local branch move only, uncommitted changes preserved — **not** a force-push, origin untouched) + write the bad sha to `state/.update_hold`, which self-update.sh refuses to re-pull until origin moves past it. Otherwise → `LOOPS_CRASHLOOP_COOLDOWN`(30min) restart backoff + escalate. Double-start guards in `dispatch.sh` (alive-pid check before claiming the pidfile) and `notify-bot.mjs` (pgrep self-check) make late-materializing duplicate spawns exit harmlessly.
    - **`bin/incident-bridge.sh`** (dispatch.sh housekeeping ≤120s) — turns engine-level failures into Backlog issues of the **engine self-improvement loop** (auto-detected: the enabled loop whose `repo` == LOOPS_HOME, e.g. `loops-improve`), which closes the full "self-fix + self-deploy" cycle: incident issue → worker fixes it (under the mission's syntax-check + core-path human-gate rules) → direct push → self-update → dispatcher re-exec. Signals filed: an orchestrator cycle failing `LOOPS_INCIDENT_FAILS`(2)+ consecutive runs (evidence = run.log tail), and supervisor `escalate`/`rollback` events. Deliberately **not** filed: `stuck` / `rework-exhausted` — those are per-issue *target-repo* problems the engine loop's worker cannot fix, and they already have first-class human surfacing. Dedup = streak-reset for cycle errors + per-signature cooldown for supervisor events, plus a global daily cap `LOOPS_INCIDENT_DAILY_MAX`(3). Creation via `bin/linear-create.mjs` (stdin description); state `state/incidents.json`; every filing is appended to the engine loop's `runs.jsonl` + Telegram-notified.
  - **Retro / hill-climbing** (config `retro:{everyCycles:N}`) — every N regular cycles dispatch.sh fires a `LOOP_MODE=retro` run (`retro-base.md`): analyzes merged-vs-closed outcomes, repeated review comments, human-gate precedents (`decisions/*.md`) and rewrites `state/learnings.md` (≤15 evidence-backed lines). Injected as `{{LEARNINGS}}` into orchestrator+worker prompts. **mission.md is never touched by the engine** — missions are human-owned; lessons live in the separate file, editable in the dashboard ⚙️ modal (with a manual "🧠 retro 실행" button / `run-retro` control action), and every update pings Telegram (comprehension-debt guard).
- **`dashboard-server.mjs`** — dependency-free Node HTTP server. `GET /api/status` aggregates each loop's `state/snapshot.json` + live cmux tabs + a 60s background `gh pr view` cache. `POST /api/control` dispatches actions (`start`, `run-now`, `reconcile`, `resolve-gate`, `start-issue`, `toggle-enabled`, `set-schedule`, `build-loop`, `create-loop`, `save-mission`, `save-config`, …) by shelling out to the same `bin/` scripts and `cmux`/`gh`. It also serves pinned static assets at `/vendor/*` (filename-whitelisted) — currently Oat UI. `dashboard.html` is the single-file UI: bespoke telemetry layout (sidebar/gauge/LED-dot/feed/cards) on top of **Oat UI** components (buttons, inputs, `.badge`, `role=alert`, `<ot-dropdown>` menus, native-title tooltips). The whole telemetry palette (dark + light via `data-theme`) is mapped onto Oat's CSS variables, so Oat components inherit the console look. Global controls (dispatcher start/pause/stop/logs + workspace settings) live in the `⚙️ 디스패처` `<ot-dropdown>`; per-loop `⚙️ 설정` vs global `워크스페이스 설정` (Linear key, Telegram bot) are separate modals.
- **`bin/notify-bot.mjs`** — optional Telegram remote bridge (zero-dep, `node:https`). One process, two loops: polls `GET /api/status` and diffs to **push** human-gate/PR/CI/cycle-error alerts to the owner's phone; long-polls Telegram `getUpdates` for **inbound** button taps / message replies / slash commands, mapping each to a `POST /api/control`. It is a full remote control, not just alerts: an interactive `/menu` (inline keyboards) navigates dispatcher→loops→tasks and exposes nearly every dashboard action — global `start`/`stop`/`pause`/`resume`/`awake`, per-loop `run-now`/`reconcile`/`loop-pause`/`toggle-enabled`, per-issue `resolve-gate`/`cancel-issue`/`cleanup-issue` — plus slash fallbacks (`/status` lists active In Progress/In Review tasks). Free-text messages that aren't slash commands or gate replies now run a **real agent**: `claude` headless (`-p`, capable model via `LOOPS_BOT_AGENT_MODEL`, default Sonnet) whose **only** toolset is the Loops control MCP server (`bin/loops-mcp.mjs`, a zero-dep stdio JSON-RPC server that thinly relays `/api/status`·`/api/session`·`/api/control` over 127.0.0.1). The agent multi-steps on its own — e.g. "seo에 'OG태그 누락 점검' 태스크 추가하고 바로 작업 시작해" → `get_status` → `create_issue{start:true}`; "요즘 CI 왜 깨져? 원인 보고 고칠 태스크 만들어" → `get_run_log`/`get_worker_screen` → several `create_issue`. `create-issue` (new `/api/control` action) creates a Linear Backlog issue in the loop's project via `issueCreate`, and with `start:true` immediately spawns a worker (mirrors dashboard `start-issue`); task-add follows the user's literal words (add-only vs add+start). The bot passes the MCP config **inline** (`--mcp-config <json>`), with `--strict-mcp-config` (ignore the user's global MCP servers) and `--disallowedTools Bash Edit Write NotebookEdit …` so the agent **cannot merge/deploy/force-push** — that's enforced structurally (no such tool exists), not by prompt. **The destructive actions `cancel_issue`/`cleanup_issue` are deliberately absent from the agent's toolset** — the agent proposes and points the user to the existing 2-tap confirm buttons / `/cancel`; it never cancels or cleans up on its own. The MCP server relays only the dashboard HTTP surface — it does **not** touch cmux — so the engine is unchanged. Auth = `TELEGRAM_CHAT_ID` lock (auto-paired on first message); token in `loops.env` (`TELEGRAM_BOT_TOKEN`, a preserved key). Launched via `loopctl bot` **or** the dashboard ⚙️ (token field + ▶ 봇 시작, backed by control actions `set-telegram`/`bot-start`/`bot-stop`; `GET /api/status` exposes `telegram:{configured,paired,running}` booleans only, never the token). **The bot never merges/deploys/force-pushes** — the destructive actions it exposes are `cancel-issue`/`cleanup-issue` (two-tap confirm); `resolve-gate` only relays the human's decision to the worker. Merging stays a human action.

## A loop's anatomy

`loops/<id>/` (gitignored — user data) contains:
- `config.json` — schema: `id, name, emoji, repo, baseRef (origin/develop), prBase (develop), branchPrefix, orchestratorWorktree, worktreePrefix, linearProjectId, linearProjectUrl, maxWorkers, backlogTarget, schedule{startAt,intervalSec}, enabled` + opt-in feedback-layer fields: `verify (bool), budget{dailyUsd}, on{ciFailure,prReview,linearNew}, retro{everyCycles}`. See `examples/*/config.json` and the README field table.
- `mission.md` — injected as `{{MISSION}}` into the orchestrator prompt; defines *what work to discover and how* (the only domain-specific logic).
- `state/` — runtime: `snapshot.json` (dashboard reads this — orchestrator writes it in STEP 4), `runs.jsonl` (append-only event feed), `run.log`, `next_fire`, `decisions/<ISSUE>.md` (human-gate resolutions), `PAUSED`, `costs.jsonl` (headless-run usage), `rework.json` (review-rework counters), `events.json` (event-poll cursors), `verify/<ISSUE>.json` (verifier verdicts), `learnings.md` (retro lessons → `{{LEARNINGS}}`), `liveness.json` (watchdog).

The Linear project **is** the state machine (dedup across runs): `Backlog → In Progress → In Review → Done/Canceled`. A "run log" tracking issue per loop gets one comment per cycle.

## Conventions that matter when editing

- **Never merge / deploy / force-push** from engine code or prompts. Workers open PRs only; merging is the human gate. Preserve this in any prompt or script change. **Exception (opt-in):** a loop with `"delivery": "direct"` in its `config.json` makes its workers push straight to `prBase` (non-force, rebase-then-push, stop on conflict) instead of opening a PR — used for personal/no-reviewer repos like the `loops-improve` self-improvement loop. Default (no `delivery` field) = `"pr"` = the safe PR flow. `render-prompt.mjs` swaps the worker's step 4–7 (`{{WORKER_DELIVERY}}`) and an orchestrator note (`{{DELIVERY_NOTE}}`) based on this flag. **force-push stays banned in both modes.**
- **No silent fallback.** Both base prompts forbid `?? default`, swallowed catches, and guessing — fix root cause. The target repos enforce this via their own CLAUDE.md/AGENTS.md (which workers/orchestrators must obey, since cwd is the *target* repo's worktree, not this one).
- **`spawn-orchestrator.sh` must stay foreground** (the caller backgrounds it). Detaching loses cmux socket access, so workers can't spawn — hence the repeated "FOREGROUND / cmux 안에서 실행" warnings.
- **cmux lazy-PTY: a tab's command only starts once the tab actually renders on screen.** `new-workspace --command` merely queues "text+Enter"; from a non-cmux caller (launchd, an external shell) — or into a window that's minimized / on another Space — the workspace object is created but no PTY exists, so the command silently waits and fires **much later** when the tab first renders (the double-dispatcher hazard). Therefore every panel spawn goes through **`bin/spawn-panel.sh`**: create → verify materialization (`read-screen`) → escalate (select → app activate → new-window+move+focus) → on final failure **close the workspace** (so nothing fires late) and exit non-zero. `loopctl start|dashboard|bot|run-now` all use it. This constraint is also why supervisor `full` revives the dispatcher via the dashboard proxy first, and why the `panels` scope runs from inside dispatch.sh rather than launchd.
- **PR URLs come only from `gh pr view --json url`** — never construct `org/repo` URLs by hand (origin may be a GitHub mirror).
- Concurrency: `in-flight = (In Progress) + (In Review)`; cap = `min(maxWorkers, LOOP_MAX_WORKERS)`. Per-loop run lock is a `/tmp/loop-<id>.lockdir` mkdir.
- **human-gate**: an issue whose body says "human-gate" is *not* implemented by a worker — it surfaces to the dashboard (🔴) for a human decision, which is written to `state/decisions/<ISSUE>.md` and injected back into the worker as an authoritative override on the next spawn.
- **Terminal-state cleanup.** Once an issue reaches a terminal state — `Done` (PR merged) or `Canceled` (PR closed / user discarded) — its cmux tab + git worktree + branch are removed automatically. This runs deterministically in `bin/run-once.sh` *after* the headless orchestrator (every cycle, all `LOOP_MODE`s) via `bin/cleanup-issue.sh <loop> <issue>` — idempotent, matches the cmux tab by title (`🛠/↩ <loop> <issue>`), gated on the worktree dir actually existing. The same helper backs the dashboard `cancel-issue` ("🗑 버리기") and a per-issue "🧹 정리" button; `bin/cleanup-loop.sh` (dashboard `delete-loop` and `loopctl cleanup <loop>`) sweeps a whole loop. **In-progress worktrees (In Progress / In Review) are still preserved** so a user can `claude --resume` them; the LLM prompts never do this cleanup (deterministic shell only); **force-push / merge / deploy stay banned.** The orchestrator always rebuilds its own worktree from `baseRef` (never the user's working tree).
- **Liveness & stall recovery is deterministic, keyed on Linear (not the stale snapshot).** Two shell loops run in `dispatch.sh` every ≤60s (both skip a loop while its `/tmp/loop-<id>.lockdir` is held, so they never race an orchestrator run):
  - **`bin/watchdog.sh`** — the in-flight set is **Linear `started`** (always fresh; the old code keyed on existing-worktrees + hourly `snapshot.json`, which missed no-worktree ghosts and lagged a full cycle). ⚠️ A cmux tab does **not** reliably die with its `--command` — the pane can drop to a bare shell, and a cmux **app restart session-restores title-only shell tabs** (dead worker looks alive → wedged false-positive + heal blocked). So liveness is process-based: `worker-run.sh` writes `state/live/<ISSUE>.pid` on start, and on exit removes it + retitles its tab **⏹** (out of the `🛠|↩` liveness regexes; cleanup-issue/loop sweep ⏹ too). Per `started` issue: *branch has any PR* (delivered — In Review or awaiting Done/Canceled reconcile) → not watchdog's concern, **except** the pidfile corpse check still runs: with resident monitors an In Review issue keeps a live 🛠 tab, and a cmux-restart corpse there would permanently block rework/heal's live-tab dedup (`close_if_corpse`). *Live tab* (no PR yet) → first, if a pidfile exists and its pid is dead, the tab is a corpse → watchdog closes it and lets the next pass heal (no pidfile → conservative, legacy behavior); else `cmux read-screen` hash; if the screen is frozen ≥ `LOOP_WEDGE_SEC` (300s) it's **`wedged`** (surfaced only — never auto-killed, to avoid killing a slow-but-fine worker). *No tab* → if a branch **open PR** exists (pr-mode) it's In Review (done, awaiting human merge → leave it); else it's a dead In Progress — if a **worktree remains** (progress) → `bin/heal-worker.sh` resumes it in place (grace → `LOOP_HEAL_MAX` attempts → `escalated`), if **no worktree** it's a ghost the watchdog does **not** re-spawn (would bypass the cap).
  - **`bin/cleanup-terminal.sh` (the reaper)** — beyond terminal cleanup, when Linear is fresh it also (4) reverts **no-worktree `started` ghosts → Backlog** via `bin/linear-move.mjs` (a ghost pins `in-flight` so `capacity = cap − in-flight` goes ≤0 and the orchestrator stops spawning — this is the usual "loop silently stops"; reverting frees the slot so the orchestrator re-spawns within cap), and (5) reclaims **Backlog-issue worktree litter** (dead worker reverted to Backlog, no live tab/PR) via `cleanup-issue.sh`. Both gated on live Linear + no open PR; `escalated` issues are vetoed. **Backlog moves are not merge/cancel — the no-merge rule is intact.**
  - The dashboard reads `state/liveness.json` and surfaces `stuck` (escalated), `wedged`, and `healing`. The orchestrator prompt now **defers** dead-worker handling to these loops (LLM is a rare backstop), so it no longer races them.

## Adding a loop

Prefer the **`create-loop` skill** (or dashboard "+ 새 loop" → `bin/build-loop.sh`), which follows `bin/loop-builder.md`: it creates a Linear project, writes `mission.md` + `config.json` under `loops/<id>/`, and leaves it `enabled:false` for human review. Manual path: copy an `examples/<...>/` dir to `loops/<id>/` and fill in repo / Linear IDs / worktree paths.
