# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

**Loops** вАФ a multi-loop autonomous-agent platform. Each "loop" is a recurring, headless Claude Code agent for one domain (SEO, dead-code cleanup, webview refactor, вА¶). A loop's **orchestrator** discovers work and fans it out; **workers** each implement one item and open a PR. **Humans merge** вАФ the engine never merges, deploys, or force-pushes.

The engine is shared; a loop differs only in its `mission.md` (what to find, how) and `config.json` (repo / Linear / schedule / concurrency).

There is **no build, test, or lint step**. The engine is **zero dependencies** вАФ `zsh` scripts + Node built-ins (`http`, `child_process`, `fs`); don't go looking for `package.json`, a test runner, or CI for this repo; there is none. **One exception, dashboard UI only:** the dashboard vendors [Oat UI](https://oat.ink) (MIT, ~10KB) as pinned static assets in `vendor/oat.min.{css,js}` вАФ served by `dashboard-server.mjs` at `/vendor/*` and loaded by `dashboard.html`. **This is still no-build** (edit `dashboard.html` вЖТ refresh; no bundler, no `npm install`, `dashboard-server.mjs` stays Node-builtin-only). To bump Oat, re-`curl` a pinned version from `unpkg.com/@knadh/oat@<ver>/` into `vendor/` (Oat is sub-v1 вАФ pin, don't float).

## Hard prerequisite: cmux

The platform is built on **cmux** (a terminal with socket control). The dispatcher and dashboard run *inside cmux panels*, and workers spawn as *live cmux tabs*. Non-cmux environments are unsupported. `claude`, `gh`, `node`, `git` must also be present.

## Commands

```sh
./install.sh                 # detect tools вЖТ write loops.env, ~/.loops-home, symlink create-loop skill
./loopctl dashboard          # dashboard at http://localhost:8422 (opens a cmux panel)
./loopctl start              # start the global dispatcher (cmux panel)
./loopctl stop|pause|resume|status
./loopctl run-now <loop-id>  # fire one orchestrator cycle for a loop immediately
```

There are no unit tests. To exercise a change, run a single orchestrator cycle directly:

```sh
LOOP_MODE=full ./bin/run-once.sh <loop-id>      # full cycle (headless claude -p), logs вЖТ loops/<id>/state/run.log
LOOP_MODE=reconcile ./bin/run-once.sh <loop-id> # only merge-cleanup + snapshot (fast)
node ./bin/render-prompt.mjs <loop-id> orchestrator   # inspect the rendered prompt without running it
```

`LOOP_MODE` is `full` (default) | `audit_only` (skip fan-out) | `reconcile` (only STEP 1 merge-cleanup + STEP 4 snapshot) | `retro` (performance analysis вЖТ update `state/learnings.md`; uses `retro-base.md`, no discovery/fan-out).

## Architecture

The runtime is a chain of processes, each launched in a fresh/empty context. **All cross-run continuity lives in the Linear ledger, not in memory.**

```
dispatch.sh (cmux panel, loops while-true)
  вФФвФА per enabled loop, on schedule вЖТ spawn-orchestrator.sh <id>   (FOREGROUND вАФ must keep cmux socket access)
       вФФвФА run-once.sh <id>   (claude -p <rendered orchestrator prompt>, headless, in the loop's orchestrator worktree)
            вФФвФА orchestrator fans out вЖТ spawn-worker.sh <id> <issue>   (one per Backlog item, up to cap)
                 вФФвФА new cmux tab runs worker-run.sh вЖТ claude <rendered worker prompt>   (live TUI, dedicated worktree+branch)
                      вФФвФА worker implements 1 issue вЖТ /gbase:go --no-review (polish+PR) вЖТ preview check вЖТ resident /gbase:monitor watch until merge/close (unattended rules; still never merges)
```

- **`bin/_common.sh`** вАФ sourced by every script. Derives `LOOPS_HOME` from its own location (`bin/`'s grandparent) and sources `loops.env`. This is why the repo works wherever it's cloned: paths are dynamic, never hardcoded.
- **`bin/render-prompt.mjs`** вАФ the templating core. Combines `bin/{orchestrator,worker,verifier,validator,retro}-base.md` (the shared engine logic) + the loop's `mission.md` + `state/learnings.md` (`{{LEARNINGS}}`, retro-accumulated lessons) + `config.json` values, substituting `{{VAR}}` tokens. Editing engine behavior = editing the `-base.md` files, not the per-loop files.
- **Feedback layer (closes the loop; all opt-in via config, all deterministic-guarded):**
  - **Resident monitor (pr-mode default).** The worker does **not** stop after opening its PR: after preview check / Linear In Review / verifier spawn, it stays in `/gbase:monitor` until the PR is MERGED/CLOSED, auto-applying mechanical CI fixes and mechanical/explicit review comments in-session (non-force push only). The worker prompt (`render-prompt.mjs` `WORKER_DELIVERY_PR` step 7) overrides the skill's interactive behavior for unattended tabs: no merge/approve/close, **no `--force-with-lease`** (conflicts resolved only via non-force `git merge origin/<prBase>`, otherwise surfaced as comments), **no AskUserQuestion** (ask-gate situations вЖТ PR reply + Linear "рЯЪІ мВђлЮМ нМРлЛ® нХДмЪФ" comment, keep watching), `--no-review` (monitor's self-review pass is AskUserQuestion-gated; the verifier is the loops-native checker), and no tab retitle (рЯЫ†/вЖ© is the engine's liveness signal). The gbase skills themselves are never modified. Watchdog's DELIVERED gate means a PR-open issue is never wedge-checked, so the idle watch causes no false positives; live-TUI cost of the resident session is unmeasured (same scope as workers). If the session dies (usage limits, cmux restart), the paths below take over вАФ graceful degradation to the pre-resident behavior.
  - **`bin/rework-worker.sh`** вАФ when a human reviewer leaves `CHANGES_REQUESTED` on an open PR **or** the verifier records a вЭМ fail verdict, respawns the worker in its **preserved worktree** (`LOOP_REWORK=1` вЖТ worker-run injects a rework block: address explicit requests only, non-force push, reply to comments, then re-enter the resident monitor watch). With the resident monitor alive this never fires (live-tab dedup) вАФ it is the **dead-session fallback** that also re-establishes the watcher. Orchestrator STEP 1 calls it for every OPEN PR unconditionally вАФ 3 built-in guards make that safe: new-feedback gate (won't refire until a review/verdict *newer* than the last rework), per-issue cap `LOOP_REWORK_MAX` (default 2, then `rework-exhausted` surfaces to dashboard/bot), live-tab dedup. Orchestrator STEP 1's direct mechanical-CI-fix is gated on the same signal: it only touches the branch when rework's output shows no live tab (avoids racing the resident session). State: `state/rework.json`.
  - **`bin/spawn-verifier.sh` + `verifier-run.sh` + `verifier-base.md`** (config `"verify": true`, pr-mode only) вАФ maker/checker split: after the worker opens a PR it spawns a **fresh-context verifier** in a detached `-vf` worktree (рЯФО tab) that grades the PR against the issue's acceptance criteria, actually runs repo checks, and posts an evidence-backed verdict (вЬЕ pass / вЪ†пЄП concerns / вЭМ fail) to PR + Linear + `state/verify/<ISSUE>.json`. The verifier **cannot edit code** вАФ Edit/Write are blocked via `--disallowedTools` (structural, not prompt). fail вЖТ rework path above. Never merges/moves Linear states.
  - **`bin/spawn-validator.sh` + `validator-run.sh` + `validator-base.md`** (config `"validate": true`; proposal-type loops вАФ see `docs/factory.md` M2) вАФ discoverer/skeptic split: after each orchestrator run, `run-once.sh` deterministically spawns a **fresh-context validator** (detached `-vd` worktree at `baseRef`, рЯІ™ tab) for every un-judged human-gate proposal issue in the snapshot. It re-verifies the proposal's evidence for real (WebFetch reproduction, repo grep, competitor URLs), runs a distilled YC-style demand diagnostic (specificity / interestвЙ†demand / status-quo-as-competitor / wedge / distribution), challenges premises against `vision.md`, must propose one cheaper alternative, and records a verdict (рЯЯҐ strengthen / рЯЯ° narrow / рЯФі reject) to Linear + `state/validate/<ISSUE>.json` вАФ surfaced next to `gate.ask` in the dashboard gate modal and as a рЯІ™ badge. Edit/Write blocked structurally; never approves/rejects (the human gate does), never moves Linear states. Skips if a verdict or a human decision (`decisions/<ISSUE>.md`) already exists.
  - **`vision.md`** (per-loop, optional, human-owned like `mission.md`) вАФ product direction (target user / north star / core assets / non-goals) injected as `{{VISION}}` into orchestrator¬Јretro¬Јvalidator prompts. Split from mission so retro can never touch direction; editable in the dashboard вЪЩпЄП modal (`save-vision`, `GET /api/mission?vision=1`). Empty/absent вЖТ block omitted entirely.
  - **`bin/event-poll.sh`** (config `on:{ciFailure,prReview,linearNew}`) вАФ dispatch.sh polls вЙ§60s and pulls `next_fire` to now on: new CI failure on `prBase`, new human review on loop PRs, new Backlog issue in the Linear project. No new execution path вАФ firing stays in the dispatcher's existing loop; every trigger is recorded in `runs.jsonl` (`event:"trigger"`). Cursors in `state/events.json` (first poll seeds silently).
  - **`bin/record-cost.mjs`** вАФ every headless run (`run-once.sh` cycles, verifier, validator) executes claude with `--output-format json`; this splits the output into the human-readable `run.log` text + a `state/costs.jsonl` usage record. `dashboard-server.mjs` aggregates it as per-loop `economics` (today/total USD, cost per merged, merge-rate, rework count); config `budget:{dailyUsd}` is a **soft cap** вАФ dispatcher skips fires while today's sum вЙ• cap (in-flight workers untouched, resets at midnight). Live-TUI worker cost is *not* measured (headless-only scope, labeled as such in the UI) вАФ this now includes the resident monitor phase, which extends each worker session's unmeasured lifetime until merge.
  - **Self-healing platform layer (process-level вАФ closes the "engine itself died" gap).** All other recovery (watchdog/reaper/self-update/event-poll) lives *inside* `dispatch.sh`'s while-loop, so the dispatcher itself was a single point of failure. Three deterministic-shell pieces close it (no merge/deploy/force-push anywhere):
    - **`bin/supervisor.sh`** вАФ one watchdog pass, two scopes split by the cmux render constraint (below). `full` (launchd agent `com.loops.supervisor`, 60s; `loopctl supervisor install|remove|status|run`) supervises **the dispatcher only**: dead вЖТ revive via вУ™ reuse an already-queued **`вП≥`-marked** рЯФБ tab (only the intentional-queue marker counts вАФ a marker-less un-materialized рЯФБ tab may be a session-restored shell whose "reuse" is a no-op; a materialized-but-still-вП≥ tab is a fired-and-died shell вЖТ closed) вС† dashboard `/api/control start` proxy вС° direct `loopctl start`, then verify the pid ~3s later. **Death vs spawn-blocked**: a "restart" only counts toward crash-loop when a real death is observed (`dispatcher.lastAlivePid` seen alive then gone, or new `dispatcher start (pid` lines in dispatcher.log beyond `dispatcher.startsSeen`); if neither signal fires the dispatcher never started (backgrounded cmux queue) вЖТ **spawn-blocked**: retry without counting/hold + rate-limited "foreground cmux" Telegram hint вАФ this is what previously mis-escalated to a 30-min hold that blocked recovery. `panels` (called from dispatch.sh housekeeping вЙ§60s) supervises **dashboard + bot** from the dispatcher's PTY context, and runs **`sweep_panels`**: each panel records its own tab at boot via `cmux identify` (`state/panel.{dispatcher,dashboard,bot}.ref`), and the sweep closes every рЯФБ/рЯУК/рЯ§Ц-titled tab that isn't the recorded live one (plus all `вПє loops *` exit-marked tabs) вАФ this is what reaps the title-only shells a cmux app restart session-restores (worker tabs have their own pidfile/watchdog path; `рЯ§Ц loop builder` and user tabs are untouchable; missing ref file or empty list-workspaces вЖТ that class is skipped, flake principle). Intent markers `state/STOPPED.{dispatcher,bot}` (written by `loopctl stop` / dashboard stop actions, cleared by every start path) distinguish crash from deliberate stop вАФ marked components are never revived. Dashboard "process alive but HTTP dead" вЖТ surfaced only, never killed (wedged philosophy). Actions append to `state/supervisor-events.jsonl`; alerts go out via **`bin/tg-notify.mjs`** (direct Telegram send, deliberately *not* through the bot вАФ the bot is itself a supervised component), rate-limited per key. State: `state/supervisor.json`; opt-outs: `LOOPS_SUPERVISE_DASHBOARD=0`.
    - **Dispatcher-only queued-spawn exception** вАФ `spawn-panel.sh` normally discards a workspace that fails to materialize (no-late-fire rule). With `SPAWN_PANEL_QUEUE_OK=1` (used *only* for the dispatcher by `loopctl start` and the dashboard `start` action) it instead leaves ONE tab queued, titled `рЯФБ loops dispatcher вП≥`, and exits 2: while the cmux app is backgrounded **no** new tab renders (escalation included), so discard-and-retry can never converge вАФ the вП≥ tab auto-fires the moment the user foregrounds cmux, and the dispatch.sh double-start guard makes any duplicate firing harmless (dispatch.sh strips the вП≥ marker and records `state/panel.dispatcher.ref` + runs one immediate panels sweep at startup, and retitles its tab `вПє loops dispatcher` on clean exit). Never use QUEUE_OK for workers/verifiers вАФ they have no double-start guard.
    - **Crash-loop guard + self-update rollback** вАФ вЙ•`LOOPS_CRASHLOOP_N`(3) dispatcher deaths within `LOOPS_CRASHLOOP_WINDOW`(600s): if the newest `state/.update_history` line (appended by self-update.sh on every ff) is <`LOOPS_ROLLBACK_WINDOW`(1h) old and HEAD == its result, the update is the suspect вЖТ `git reset --keep <prev-sha>` (local branch move only, uncommitted changes preserved вАФ **not** a force-push, origin untouched) + write the bad sha to `state/.update_hold`, which self-update.sh refuses to re-pull until origin moves past it. Otherwise вЖТ `LOOPS_CRASHLOOP_COOLDOWN`(30min) restart backoff + escalate. Double-start guards in `dispatch.sh` (alive-pid check before claiming the pidfile) and `notify-bot.mjs` (pgrep self-check) make late-materializing duplicate spawns exit harmlessly.
    - **`bin/incident-bridge.sh`** (dispatch.sh housekeeping вЙ§120s) вАФ turns engine-level failures into Backlog issues of the **engine self-improvement loop** (auto-detected: the enabled loop whose `repo` == LOOPS_HOME, e.g. `loops-improve`), which closes the full "self-fix + self-deploy" cycle: incident issue вЖТ worker fixes it (under the mission's syntax-check + core-path human-gate rules) вЖТ direct push вЖТ self-update вЖТ dispatcher re-exec. Signals filed: an orchestrator cycle failing `LOOPS_INCIDENT_FAILS`(2)+ consecutive runs (evidence = run.log tail), and supervisor `escalate`/`rollback` events. Deliberately **not** filed: `stuck` / `rework-exhausted` / **`auth/quota abort`** вАФ the first two are per-issue *target-repo* problems the engine loop's worker cannot fix (and already have first-class human surfacing); `auth/quota abort` (a `run.log` tail matching `ACCOUNT_ABORT_RE` вАФ weekly-limit / logged-out / invalid-key / low-credit) is a *claude account* state with no engine code to fix, so it skips **only** the cycle-error filing **without** consuming the streak/filed cursor (a genuine crash streak after the signature clears is still filed normally) вАФ this is the basis for treating such aborts as environment noise, not engine defects. Dedup = streak-reset for cycle errors + per-signature cooldown `LOOPS_INCIDENT_COOLDOWN`(86400s) for supervisor events, plus a global daily cap `LOOPS_INCIDENT_DAILY_MAX`(3). Creation via `bin/linear-create.mjs` (stdin description); state `state/incidents.json`; every filing is appended to the engine loop's `runs.jsonl` + Telegram-notified.
  - **Retro / hill-climbing** (config `retro:{everyCycles:N}`) вАФ every N regular cycles dispatch.sh fires a `LOOP_MODE=retro` run (`retro-base.md`): analyzes merged-vs-closed outcomes, repeated review comments, human-gate precedents (`decisions/*.md`) and rewrites `state/learnings.md` (вЙ§15 evidence-backed lines). Injected as `{{LEARNINGS}}` into orchestrator, worker, and validator prompts. **mission.md is never touched by the engine** вАФ missions are human-owned; lessons live in the separate file, editable in the dashboard вЪЩпЄП modal (with a manual "рЯІ† retro мЛ§нЦЙ" button / `run-retro` control action), and every update pings Telegram (comprehension-debt guard).
- **`dashboard-server.mjs`** вАФ dependency-free Node HTTP server. `GET /api/status` aggregates each loop's `state/snapshot.json` + live cmux tabs + a 60s background `gh pr view` cache. `POST /api/control` dispatches actions (`start`, `run-now`, `reconcile`, `resolve-gate`, `start-issue`, `toggle-enabled`, `set-schedule`, `build-loop`, `create-loop`, `save-mission`, `save-config`, вА¶) by shelling out to the same `bin/` scripts and `cmux`/`gh`. It also serves pinned static assets at `/vendor/*` (filename-whitelisted) вАФ currently Oat UI. `dashboard.html` is the single-file UI: bespoke telemetry layout (sidebar/gauge/LED-dot/feed/cards) on top of **Oat UI** components (buttons, inputs, `.badge`, `role=alert`, `<ot-dropdown>` menus, native-title tooltips). The whole telemetry palette (dark + light via `data-theme`) is mapped onto Oat's CSS variables, so Oat components inherit the console look. Global controls (dispatcher start/pause/stop/logs + workspace settings) live in the `вЪЩпЄП лФФмК§нМ®м≤Ш` `<ot-dropdown>`; per-loop `вЪЩпЄП мД§м†Х` vs global `мЫМнБђмК§нОШмЭімК§ мД§м†Х` (Linear key, Telegram bot) are separate modals.
- **`bin/notify-bot.mjs`** вАФ optional Telegram remote bridge (zero-dep, `node:https`). One process, two loops: polls `GET /api/status` and diffs to **push** human-gate/PR/CI/cycle-error alerts to the owner's phone; long-polls Telegram `getUpdates` for **inbound** button taps / message replies / slash commands, mapping each to a `POST /api/control`. It is a full remote control, not just alerts: an interactive `/menu` (inline keyboards) navigates dispatcherвЖТloopsвЖТtasks and exposes nearly every dashboard action вАФ global `start`/`stop`/`pause`/`resume`/`awake`, per-loop `run-now`/`reconcile`/`loop-pause`/`toggle-enabled`, per-issue `resolve-gate`/`cancel-issue`/`cleanup-issue` вАФ plus slash fallbacks (`/status` lists active In Progress/In Review tasks). Free-text messages that aren't slash commands or gate replies now run a **real agent**: `claude` headless (`-p`, capable model via `LOOPS_BOT_AGENT_MODEL`, default Sonnet) whose **only** toolset is the Loops control MCP server (`bin/loops-mcp.mjs`, a zero-dep stdio JSON-RPC server that thinly relays `/api/status`¬Ј`/api/session`¬Ј`/api/control` over 127.0.0.1). The agent multi-steps on its own вАФ e.g. "seoмЧР 'OGнГЬкЈЄ лИДлЭљ м†Рк≤А' нГЬмК§нБђ мґФк∞АнХШк≥† л∞Фл°Ь мЮСмЧЕ мЛЬмЮСнХі" вЖТ `get_status` вЖТ `create_issue{start:true}`; "мЪФм¶Ш CI мЩЬ кє®м†Є? мЫРмЭЄ л≥ік≥† к≥†мє† нГЬмК§нБђ лІМлУ§мЦі" вЖТ `get_run_log`/`get_worker_screen` вЖТ several `create_issue`. `create-issue` (new `/api/control` action) creates a Linear Backlog issue in the loop's project via `issueCreate`, and with `start:true` immediately spawns a worker (mirrors dashboard `start-issue`); task-add follows the user's literal words (add-only vs add+start). ## Product layer (optional, above loops)

`products/<id>/product.json` (gitignored вАФ user data) groups multiple loops onto **one product = one Linear project** (e.g. PM loop on `linearLabel:"Feature"` + bug loop on `"Bug"`). Two mechanisms, both backward-compatible (no `product` field вЖТ кЄ∞м°і лЛ®лПЕ л£®нФД лПЩмЮС кЈЄлМАл°Ь):
- **Config inheritance** вАФ a loop config with `"product": "<id>"` inherits `repo, baseRef, prBase, claudeCmd, linearProjectId, linearProjectUrl` from product.json (loop value always wins; whitelist only вАФ `bin/loop-config.mjs`, mirrored inline in `_common.sh` `cfgval`; consumers: cfgval(shell м†ДлґА)¬Јrender-prompt¬Јdashboard-server). Loop-specific settings (schedule/cap/label/worktrees/enabled) stay in the loop config.
- **`bin/triage.sh` (product-level classifier)** вАФ dispatch.sh polls вЙ§60s per product: unlabeled inbox issues (no route label; `bin/linear-inbox.mjs`) are classified by a cheap headless claude call (`triage.routes` in product.json; MCP 0 ¬Ј tool 0 вАФ the LLM only *picks* `{"label","reason"}`, the deterministic `bin/linear-label.mjs` applies it and rejects labels not in routes) вЖТ the moment the label lands, that label's loop event-poll (`on.linearNew`, label-scoped) pulls an immediate cycle. Per-issue 3-attempt cap вЖТ give-up comment ("лЭЉл≤® мІБм†С мІАм†Х"). Dedup is Linear itself (лЭЉл≤® лґАмЮђ = лѓЄлґДл•Ш); local state (`products/<id>/state/triage.json`) holds attempts only. No merge/state-moves/issue-creation вАФ label+comment only.

## A loop's anatomy

`loops/<id>/` (gitignored вАФ user data) contains:
- `config.json` вАФ schema: `id, name, emoji, repo, baseRef (origin/develop), prBase (develop), branchPrefix, orchestratorWorktree, worktreePrefix, linearProjectId, linearProjectUrl, maxWorkers, backlogTarget, schedule{startAt,intervalSec}, enabled, claudeCmd, delivery` + opt-in feedback-layer fields: `verify (bool), validate (bool), budget{dailyUsd}, on{ciFailure,prReview,linearNew}, retro{everyCycles}` + shared-project/drain fields: `linearLabel (str вАФ нХШлВШмЭШ Linear нФДл°Ьм†ЭнКЄл•Љ лЭЉл≤®л°Ь лВШлИ† мЧђлЯђ л£®нФДк∞А к≥µмЬ†; мІАм†Х мЛЬ м°∞нЪМ¬Јл∞Ькµі¬Јfan-out¬Јм†Хл¶ђ¬ЈмЭіл≤§нКЄк∞А кЈЄ лЭЉл≤®л°Ь мК§мљФнФДлРШк≥† мГИ мЭімКИмЧР лЭЉл≤® лґАмЧђ; `linear-states.mjs`мЭШ мШµмЕШ мЭЄмЮРл°Ь к≤∞м†Хл°† мЄµмЧР, `{{LINEAR_LABEL_NOTE}}`л°Ь нФДл°ђнФДнКЄ мЄµмЧР м£ЉмЮЕ; лєДл©і нФДл°Ьм†ЭнКЄ м†Дм≤і=кЄ∞м°і лПЩмЮС), drain{discoverySec} (bool|obj вАФ "мМУмЭіл©і к≥ДмЖН, лєДл©і м°∞мЪ©": `dispatch.sh`к∞А мК§мЉАм§Д л∞ЬмВђл•Љ к∞ТмЛЉ Linear м≤інБђ(`linear-drain-check.mjs` вАФ run-log¬Јhuman-gate мЭімКИлКФ лУЬл†ИмЭЄ лґИк∞Ал°Ь м†ЬмЩЄ)л°Ь к≤МмЭінКЄ вАФ лУЬл†ИмЭЄ к∞АлК• backlog>0 & in-flight<cap лШРлКФ л∞Ькµі м£ЉкЄ∞ лПДлЮШмЭЉ лХМлІМ мВђмЭінБі, мХДлЛИл©і мК§нВµвЖТidle нЖ†нБ∞лєДмЪ© 0)`. The Linear project **is** the state machine (dedup across runs): `Backlog вЖТ In Progress вЖТ In Review вЖТ Done/Canceled`. A "run log" tracking issue per loop gets one comment per cycle.

## Conventions that matter when editing

- **Never merge / deploy / force-push** from engine code or prompts. Workers open PRs only; merging is the human gate. Preserve this in any prompt or script change. **Exception (opt-in):** a loop with `"delivery": "direct"` in its `config.json` makes its workers push straight to `prBase` (non-force, rebase-then-push, stop on conflict) instead of opening a PR вАФ used for personal/no-reviewer repos like the `loops-improve` self-improvement loop. Default (no `delivery` field) = `"pr"` = the safe PR flow. `render-prompt.mjs` swaps the worker's step 4вАУ7 (`{{WORKER_DELIVERY}}`) and an orchestrator note (`{{DELIVERY_NOTE}}`) based on this flag. **force-push stays banned in both modes.**
- **No silent fallback.** Both base prompts forbid `?? default`, swallowed catches, and guessing вАФ fix root cause. The target repos enforce this via their own CLAUDE.md/AGENTS.md (which workers/orchestrators must obey, since cwd is the *target* repo's worktree, not this one).
- **`spawn-orchestrator.sh` must stay foreground** (the caller backgrounds it). Detaching loses cmux socket access, so workers can't spawn вАФ hence the repeated "FOREGROUND / cmux мХИмЧРмДЬ мЛ§нЦЙ" warnings.
- **cmux lazy-PTY: a tab's command only starts once the tab actually renders on screen.** `new-workspace --command` merely queues "text+Enter"; from a non-cmux caller (launchd, an external shell) вАФ or into a window that's minimized / on another Space вАФ the workspace object is created but no PTY exists, so the command silently waits and fires **much later** when the tab first renders (the double-dispatcher hazard). Therefore every panel spawn goes through **`bin/spawn-panel.sh`**: create вЖТ verify materialization (`read-screen`) вЖТ escalate (select вЖТ app activate вЖТ new-window+move+focus) вЖТ on final failure **close the workspace** (so nothing fires late) and exit non-zero. `loopctl start|dashboard|bot|run-now` all use it, and вАФ since GOD-59 вАФ so does `spawn-worker.sh` (workers spawn via spawn-panel too, so an un-materialized рЯЫ† tab can't wedge an In Progress slot), but **without** `SPAWN_PANEL_QUEUE_OK` вАФ workers have no double-start guard, so the queued-spawn exception stays dispatcher-only. This constraint is also why supervisor `full` revives the dispatcher via the dashboard proxy first, and why the `panels` scope runs from inside dispatch.sh rather than launchd.
- **PR URLs come only from `gh pr view --json url`** вАФ never construct `org/repo` URLs by hand (origin may be a GitHub mirror).

- Concurrency: `in-flight = (In Progress) + (In Review)`; cap = `min(maxWorkers, LOOP_MAX_WORKERS)`. Per-loop run lock is a `/tmp/loop-<id>.lockdir` mkdir.
- **human-gate**: an issue whose body says "human-gate" is *not* implemented by a worker вАФ it surfaces to the dashboard (рЯФі) for a human decision, which is written to `state/decisions/<ISSUE>.md` and injected back into the worker as an authoritative override on the next spawn.
- **Terminal-state cleanup.** Once an issue reaches a terminal state вАФ `Done` (PR merged) or `Canceled` (PR closed / user discarded) вАФ its cmux tab + git worktree + branch are removed automatically. This runs deterministically in `bin/run-once.sh` *after* the headless orchestrator (every cycle, all `LOOP_MODE`s) via `bin/cleanup-issue.sh <loop> <issue>` вАФ idempotent, matches the cmux tab by title (`рЯЫ†/вЖ© <loop> <issue>`), gated on the worktree dir actually existing. The same helper backs the dashboard `cancel-issue` ("рЯЧС л≤Дл¶ђкЄ∞") and a per-issue "рЯІє м†Хл¶ђ" button; `bin/cleanup-loop.sh` (dashboard `delete-loop` and `loopctl cleanup <loop>`) sweeps a whole loop. **In-progress worktrees (In Progress / In Review) are still preserved** so a user can `claude --resume` them; the LLM prompts never do this cleanup (deterministic shell only); **force-push / merge / deploy stay banned.** The orchestrator always rebuilds its own worktree from `baseRef` (never the user's working tree).
- **Liveness & stall recovery is deterministic, keyed on Linear (not the stale snapshot).** Two shell loops run in `dispatch.sh` every вЙ§60s (both skip a loop while its `/tmp/loop-<id>.lockdir` is held, so they never race an orchestrator run):
  - **`bin/watchdog.sh`** вАФ the in-flight set is **Linear `started`** (always fresh; the old code keyed on existing-worktrees + hourly `snapshot.json`, which missed no-worktree ghosts and lagged a full cycle). вЪ†пЄП A cmux tab does **not** reliably die with its `--command` вАФ the pane can drop to a bare shell, and a cmux **app restart session-restores title-only shell tabs** (dead worker looks alive вЖТ wedged false-positive + heal blocked). So liveness is process-based: `worker-run.sh` writes `state/live/<ISSUE>.pid` on start, and on exit removes it + retitles its tab **вПє** (out of the `рЯЫ†|вЖ©` liveness regexes; cleanup-issue/loop sweep вПє too). Per `started` issue: *branch has any PR* (delivered вАФ In Review or awaiting Done/Canceled reconcile) вЖТ not watchdog's concern, **except** the pidfile corpse check still runs: with resident monitors an In Review issue keeps a live рЯЫ† tab, and a cmux-restart corpse there would permanently block rework/heal's live-tab dedup (`close_if_corpse`). *Live tab* (no PR yet) вЖТ first, if a pidfile exists and its pid is dead, the tab is a corpse вЖТ watchdog closes it and lets the next pass heal (since GOD-59, the **no-pidfile** case is also reclaimed: no pidfile + worktree present + 0 commits ahead of `baseRef` + `CORPSE_PASSES` consecutive empty read-screens вЖТ an un-materialized corpse вЖТ `close_if_dead_shell` closes the tab, next pass heals; any pidfile / commit / non-empty screen resets it, so a slow-but-fine worker is never touched); else `cmux read-screen` hash; if the screen is frozen вЙ• `LOOP_WEDGE_SEC` (300s) it's **`wedged`** (surfaced only вАФ never auto-killed, to avoid killing a slow-but-fine worker). *No tab* вЖТ if a branch **open PR** exists (pr-mode) it's In Review (done, awaiting human merge вЖТ leave it); else it's a dead In Progress вАФ if a **worktree remains** (progress) вЖТ `bin/heal-worker.sh` resumes it in place (grace вЖТ `LOOP_HEAL_MAX` attempts вЖТ `escalated`), if **no worktree** it's a ghost the watchdog does **not** re-spawn (would bypass the cap).
  - **`bin/cleanup-terminal.sh` (the reaper)** вАФ beyond terminal cleanup, when Linear is fresh it also (4) reverts **no-worktree `started` ghosts вЖТ Backlog** via `bin/linear-move.mjs` (a ghost pins `in-flight` so `capacity = cap вИТ in-flight` goes вЙ§0 and the orchestrator stops spawning вАФ this is the usual "loop silently stops"; reverting frees the slot so the orchestrator re-spawns within cap), and (5) reclaims **Backlog-issue worktree litter** (dead worker reverted to Backlog, no live tab/PR) via `cleanup-issue.sh`. Both gated on live Linear + no open PR; `escalated` issues are vetoed. **Backlog moves are not merge/cancel вАФ the no-merge rule is intact.**
  - The dashboard reads `state/liveness.json` and surfaces `stuck` (escalated), `wedged`, and `healing`. The orchestrator prompt now **defers** dead-worker handling to these loops (LLM is a rare backstop), so it no longer races them.
