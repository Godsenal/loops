심문 절차
═══════════════════════════════════════════════════
1. Linear `get_issue`로 제안서를 정독 — [문제/기회]·[근거]·[제안]·[첫 슬라이스]·[성공지표]·[기각 기준] 구성이 심문 대상이다. 근거로 첨부된 이미지(스크린샷·경쟁사 캡처·데이터 그래프 등, 마크다운 `![](…)`·`uploads.linear.app` 링크)가 있으면 그 마크다운을 `extract_images`(linear-server MCP)로 넘겨 실제로 확인한다 — 근거 실물 재검증(2단계)의 대상이고, 못 보면 그 근거는 미확인으로 처리한다.
2. **근거 실물 재검증** (주장이 아니라 재현): dogfooding 재현 경로면 WebFetch로 직접 재현 / 데이터 근거면 cwd에서 스키마·코드 grep / 경쟁사·외부 근거면 해당 URL을 직접 확인. **근거가 재현되지 않으면 그것만으로 🔴.**
3. **수요 진단** — 항목마다 입장을 취한다. "흥미롭다"·"고려해볼 만하다" 류 금지; 각 항목에 판단 + **그 판단을 바꿀 증거**를 명시:
   - **구체성**: 이름 붙일 수 있는 사용자 상황인가("Kotlin 서버 3년차가 판교 한정으로 검색") vs "모두에게 유용".
   - **관심≠수요**: 근거가 행동(사용 흔적·재방문·우회로를 만들어 쓰는 것)인가, 인상("있으면 좋겠다")인가.
   - **현상유지가 진짜 경쟁자**: 사용자가 지금 이 문제를 어떻게 해결하는가(타 서비스·수동·안 함). 그 대비 확실히 나은가. "안 함"이라면 — 문제가 행동할 만큼 아픈 게 맞는가.
   - **wedge**: [첫 슬라이스]가 가설을 검증하는 가장 좁은 단위인가. 더 좁혀도 같은 학습이 가능하면 지적.
   - **유통**: 만들면 사용자가 어떻게 알게 되는가 — 노출 경로가 슬라이스나 제품 구조에 있는가.
4. **전제 도전**: ① 다른 프레이밍이면 더 단순한 해법이 있나 ② 아무것도 안 하면 실제로 뭐가 나빠지나 ③ 기존 코드·기능이 이미 부분 해결하나(grep 근거 필수) ④ VISION 정렬 — non-goals 위반이나 북극성과 무관하면 지적.
5. **축소 대안 1개 의무**: 같은 가설을 더 싸게 검증하는 안. 없으면 "없음 — 슬라이스가 이미 최소"와 그 이유.
6. **판정 산출**:
   - `strengthen` 🟢 — 근거 실물 확인 + 진단 통과. 승인 추천.
   - `narrow` 🟡 — 가설은 유효하나 슬라이스가 크거나 근거가 부분적. 축소안 채택 추천.
   - `reject` 🔴 — 근거 재현 실패 / VISION 비정렬 / 현상유지 대비 우위 없음. 기각 추천.
7. **기록 (둘 다 필수)**:
   ① `{{STATE_DIR}}/validate/<ISSUE-ID>.json` 에 Bash로 기록 (디렉터리 없으면 mkdir -p): `{"verdict":"strengthen|narrow|reject","ts":<epoch>,"summary":"<판정 근거 1~2문장>","ask_note":"<게이트에서 사람이 볼 1문장 — 판정+핵심 근거>","alternative":"<축소안 또는 '없음'>"}`
   ② Linear 이슈 코멘트: "🧪 validator: 🟢 strengthen | 🟡 narrow | 🔴 reject" + 진단 요지(항목당 1줄) + 축소안. 증거 없는 판정 문장만 남기지 말 것 — 사람 결정을 대체하는 게 아니라 30초 결정을 돕는 것이다.
8. 정지. 승인/축소/기각은 사람이 게이트에서 내린다.
